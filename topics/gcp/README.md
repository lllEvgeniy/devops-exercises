## Вопросы

<a id="global-infrastructure"></a>

### Глобальная инфраструктура

<details>
<summary>Объясните каждое из следующих

  * Зона
  * Регион</summary><br><b>

Регионы GCP — это центры обработки данных, расположенные в разных географических точках по всему миру.<br>

В каждом регионе есть несколько изолированных мест, известных как Зоны. Каждая зона представляет собой один или несколько дата-центров с резервной сетью, возможностью подключения и электропитанием. Несколько зон обеспечивают высокую доступность в случае выхода из строя одной из них.

</b></details>

<details>
<summary>Правда или ложь? Каждый регион GCP полностью изолирован от других регионов GCP.</summary><br><b>

Неверно. Регионы — это отдельные географические «домены» и зоны отказа, но не «абсолютная изоляция»: есть глобальные сервисы и сетевая связность между регионами, общие API и практики управления.

</b></details>

<details>
<summary>Какие соображения следует учитывать при выборе региона GCP для запуска нового приложения?</summary><br><b>

* Доступность услуг: не все услуги (и все их функции) доступны во всех регионах.
* Снижение задержки: развертывание приложения в регионе, близком к клиентам.
* Соответствие: в некоторых странах действуют более строгие правила и требования, например, обеспечение того, чтобы данные оставались в пределах границ страны или региона. В этом случае для запуска приложения можно использовать только определенный регион.
* **Цены**: цены в разных регионах могут различаться, поэтому цена на одну и ту же услугу в разных регионах может отличаться.

</b></details>

<details>
<summary>Правда или ложь? Все сервисы GCP доступны во всех регионах.</summary><br><b>

Неверно. Вы можете увидеть [здесь](https://cloud.google.com/about/locations), какие продукты/услуги доступны в каждом регионе.

</b></details>

<a id="resource-hierarchy"></a>

### Иерархия ресурсов

<details>
<summary>Объясните иерархию ресурсов в GCP</summary><br><b>

Организация
  Папка
    Проект
      Ресурсы

* Организации - Компания
* Папка — обычно для отделов, команд, продуктов и т. д.
* Проект — это могут быть разные проекты или один и тот же проект, но в разных средах (разработка, постановка, производство).
* Ресурсы – фактические сервисы GCP (вычисления, механизм приложений, хранилище и т. д.).

</b></details>

<details>
<summary>Правда или ложь? В проекте вы можете иметь одну или несколько организаций.</summary><br><b>

Неверно. Все совсем наоборот. Сначала есть организация, а в рамках организации вы можете иметь одну или несколько папок с одним или несколькими проектами.

</b></details>

<details>
<summary>Правда или ложь? Ресурс должен быть связан хотя бы с одним проектом.</summary><br><b>

Верно. Ресурс в GCP всегда принадлежит **одному** проекту; «вне проекта» существовать не может.

</b></details>

<details>
<summary>Правда или ложь? Название проекта должно быть глобально уникальным.</summary><br><b>

Верно.

</b></details>

<a id="iam-and-roles"></a>

### IAM и роли

<details>
<summary>Объясните роли и разрешения</summary><br><b>

Роль — это инкапсуляция набора разрешений. Например, роль «владельца» имеет более 3000 назначенных разрешений для различных компонентов и служб GCP.

</b></details>

<details>
<summary>Правда или ложь? Разрешительная родительская политика всегда отменяет ограничительную дочернюю политику.</summary><br><b>

Неверно. Формулировка «всегда отменяет» некорректна: в GCP действуют правила наследования и приоритета политик (в т.ч. **deny** и организационные ограничения), и более строгие/ограничивающие правила нельзя «обойти» просто «разрешительной» родительской политикой во всех случаях.

</b></details>

<a id="labels-and-tags"></a>

### Ярлыки и теги

<details>
<summary>Что такое ярлыки?</summary><br><b>

Вы можете рассматривать метки в GCP как стикеры, которые вы прикрепляете к различным ресурсам GCP. Это упрощает, например, поиск конкретных ресурсов (например, применение метки «веб-приложение» и поиск всех ресурсов, которые каким-либо образом связаны с «веб-приложением»).

</b></details>

<details>
<summary>Можете ли вы привести несколько примеров использования меток в GCP?</summary><br><b>

* Местоположение (центр затрат)
* Проект (или среда, папка и т. д.)
* Тип услуги
* Владелец сервиса
* Тип приложения
* Владелец приложения

</b></details>

<details>
<summary>Что такое сетевые теги и чем они отличаются от меток?</summary><br><b>

Как следует из названия, сетевые теги можно применять только к сетевым ресурсам.
Хотя метки не влияют на ресурсы, к которым они применяются, сетевые теги влияют на ресурсы (например, доступ к брандмауэру и сетевые маршруты).

</b></details>

<a id="другое"></a>

### Другое

<details>
<summary>Расскажите мне, что вы знаете о сетях GCP?</summary><br><b>

Сеть Virtual Private Cloud (VPC) — это виртуальная версия физической сети, реализованная во внутренней сети Google. VPC — это глобальный ресурс в GCP.
Подсети (subnets) являются региональными ресурсами, т. е. подсети могут создаваться внутри регионов.

VPC создаются в 2-х режимах,

1. Автоматический режим VPC. Одна подсеть в каждом регионе автоматически создается GCP при создании VPC.

2. Пользовательский режим VPC. Подсети автоматически не создаются. Этот тип сети предоставляет пользователям полный контроль над созданием подсетей.

</b></details>

<details>
<summary>Объяснение облачных функций</summary><br><b>

Google Cloud Functions — это бессерверная среда выполнения для создания и подключения облачных сервисов. С помощью облачных функций вы пишете простые одноцелевые функции, которые привязываются к событиям, исходящим из вашей облачной инфраструктуры и служб. Ваша функция срабатывает при возникновении наблюдаемого события.

</b></details>

<details>
<summary>Что такое облачное хранилище данных?</summary><br><b>

Cloud Datastore — это бессхемное хранилище данных NoSQL в облаке Google. Приложения могут использовать хранилище данных для запроса ваших данных с помощью SQL-запросов, поддерживающих фильтрацию и сортировку. Datastore реплицирует данные в нескольких центрах обработки данных, что обеспечивает высокий уровень доступности чтения/записи.

</b></details>

<details>
<summary>Для чего используются сетевые теги?</summary><br><b>

Сетевые теги позволяют применять правила и маршруты брандмауэра к определенному экземпляру или набору экземпляров. Вы делаете правило брандмауэра применимым к конкретным экземплярам, ​​используя целевые теги и исходные теги.

</b></details>

<details>
<summary>Что такое журналы потока? Где они включены?</summary><br><b>

Журналы потоков VPC записывают выборку сетевых потоков, отправленных и полученных экземплярами виртуальных машин, включая экземпляры, используемые в качестве узлов Google Kubernetes Engine. Эти журналы можно использовать для мониторинга сети, криминалистики, анализа безопасности в реальном времени и оптимизации расходов.

Включить журналы потоков

1. Откройте сеть VPC в консоли GCP.

2. Щелкните имя подсети. 

3. Нажмите кнопку ИЗМЕНИТЬ.

4. Включите журналы потока.

5. Нажмите «Сохранить».

</b></details>

<a id="anthos"></a>

### Anthos

<details>
<summary>Что такое Anthos?</summary><br><b>

Это управляемая **платформа приложений** для крупных организаций: модернизация и **согласованность** устаревших приложений в гибридном или мультиоблачном окружении. Ключевые идеи:

* Управляемый → клиенту не нужно беспокоиться об интеграции базового ПО, достаточно включить API.
* платформа приложений -> состоит из инструментов с открытым исходным кодом, таких как K8s, Knative, Istio и Tekton.
* Предприятия -> обычно это организации со сложными потребностями.
* Согласованность -> декларативно инициировать одни и те же политики для безопасного запуска в любом месте, например, локально, GCP или других облаках (AWS или Azure).

Интересный факт: **Anthos** (ἄνθος) — «цветок» по-гречески; метафора «цветения» в гибридном/мультиоблачном саду.

</b></details>

<details>
<summary>Перечислите технические компоненты, из которых состоит Anthos.</summary><br><b>

* Управление инфраструктурой — Google Kubernetes Engine (GKE).
* Управление кластером — GKE, Ingress для Anthos
* Управление услугами — Anthos Service Mesh.
* Применение политик — Anthos Config Management, Anthos Enterprise Data Protection, Policy Controller.
* Развертывание приложений — инструменты CI/CD, такие как Cloud Build, GitLab.
* Разработка приложений — Cloud Code

</b></details>

<details>
<summary>Какова основная вычислительная среда Anthos, позволяющая легко управлять развертыванием рабочих нагрузок?</summary><br><b>

* Google Kubernetes Engine (GKE)

</b></details>

<details>
<summary>Как Anthos обрабатывает плоскость управления и компоненты узла для GKE?</summary><br><b>

В GCP API-сервер Kubernetes — единственный компонент плоскости управления, доступный клиентам, в то время как **Compute Engine** управляет **экземплярами** в проекте.

</b></details>

<details>
<summary>Какие варианты балансировки нагрузки доступны?</summary><br><b>

* Балансировка сетевой нагрузки для L4 и балансировка нагрузки HTTP(S) для L7 — управляемые службы без необходимости в дополнительной «самодельной» конфигурации балансировщика.
* Ingress для Anthos — единый балансировщик для приложения, работающего в **нескольких кластерах GKE**.

</b></details>

<details>
<summary>Можете ли вы развернуть Anthos на AWS?</summary><br><b>

* Да, Anthos на AWS теперь общедоступный. Подробнее читайте [здесь](https://cloud.google.com/anthos/gke/docs/aws).

</b></details>

<details>
<summary>Перечислите и объясните возможности корпоративной безопасности, предоставляемые Anthos.</summary><br><b>

* Безопасность плоскости управления — GCP управляет и поддерживает плоскость управления K8s «из коробки». Пользователь может защитить API-сервер, используя основные авторизованные сети и частные кластеры. Они позволяют пользователю отключить доступ к общедоступному IP-адресу, назначив мастеру частный IP-адрес.
* Безопасность узла. По умолчанию рабочие нагрузки предоставляются на экземплярах вычислительного ядра, использующих ОС Google, оптимизированную для контейнеров. Эта операционная система реализует заблокированный брандмауэр, ограниченные учетные записи пользователей с отключенным root и файловую систему, доступную только для чтения. Существует еще один вариант включения GKE Sandbox для более строгой изоляции в сценариях развертывания с несколькими арендаторами.
* Сетевая безопасность. В созданном кластере VPC Anthos GKE использует мощную программно-определяемую сеть, которая обеспечивает простую связь между подами. Сетевые политики позволяют блокировать входящие и исходящие соединения в заданном пространстве имен. Фильтрацию также можно реализовать для входящего трафика с балансировкой нагрузки для служб, требующих внешнего доступа, путем предоставления диапазонов IP-адресов CIDR из белого списка.
* Безопасность рабочих нагрузок. При выполнении рабочих нагрузок с ограниченными привилегиями политики безопасности Docker AppArmor по умолчанию применяются ко всем подам Kubernetes. Идентификатор рабочей нагрузки для Anthos GKE соответствует учетным записям служб Kubernetes с открытым исходным кодом и разрешениями учетной записи службы GCP.
* Ведение журнала аудита. Администраторам предоставляется возможность сохранять, запрашивать, обрабатывать и оповещать о событиях в развернутых средах.

</b></details>

<details>
<summary>Как рабочие нагрузки, развернутые в локальных кластерах Anthos GKE, могут безопасно подключаться к сервисам Google Cloud?</summary><br><b>

* Виртуальная частная сеть Google Cloud (Cloud VPN) — для безопасной сети.
* Служба управления ключами Google Cloud (Cloud KMS) — для управления ключами.

</b></details>

<details>
<summary>Какова конфигурация островного режима в отношении сети в Anthos GKE, развернутой локально?</summary><br><b>

* Это когда поды могут напрямую взаимодействовать друг с другом внутри кластера, но не могут быть доступны извне кластера, образуя «островок» внутри сети, не подключенный к внешней сети.

</b></details>

<details>
<summary>Объясните управление конфигурациями Anthos</summary><br><b>

Это основной компонент стека Anthos, который предоставляет операторам платформ, услуг и безопасности единый унифицированный подход к управлению несколькими кластерами, охватывающий как локальные, так и облачные среды. Он внимательно следует лучшим практикам K8, отдавая предпочтение декларативным подходам императивным операциям, активно отслеживает состояние кластера и применяет желаемое состояние, как определено в Git. Он включает в себя три ключевых компонента:

1. Импортер, который читает из центрального репозитория Git.
2. Компонент, синхронизирующий сохраненные данные конфигурации с объектами K8s.
3. Компонент, который отслеживает расхождение между желаемой и фактической конфигурациями кластера с возможностью согласования при необходимости.

</b></details>

<details>
<summary>Чем помогает Anthos Config Management?</summary><br><b>

Он следует общепринятым современным практикам разработки программного обеспечения, что делает конфигурацию кластера, управление и изменения политик проверяемыми, обратимыми и версионируемыми, легко обеспечивая управление ИТ и унифицируя управление ресурсами в организации.

</b></details>

<details>
<summary>Что такое Anthos Service Mesh?</summary><br><b>

* Это набор инструментов, которые помогают отслеживать и управлять развернутыми службами Anthos всех форм и размеров, независимо от того, работают ли они в облачных, гибридных или мультиоблачных средах. Он использует API-интерфейсы и основные компоненты Istio, легко настраиваемой платформы Service Mesh с открытым исходным кодом.

</b></details>

<details>
<summary>Описать два основных компонента Anthos Service Mesh.</summary><br><b>

1. Плоскость данных — она состоит из набора распределенных прокси, которые являются посредниками всего входящего и исходящего сетевого трафика между отдельными службами, которые настраиваются с использованием централизованной плоскости управления и открытого API.
2. Плоскость управления — это полностью управляемое предложение за пределами кластеров Anthos GKE, позволяющее упростить управление и обеспечить максимально возможную доступность.

</b></details>

<details>
<summary>Каковы компоненты управляемой плоскости управления Anthos Service Mesh?</summary><br><b>

1. Директор трафика — это полностью управляемая плоскость управления трафиком сервисной сетки GCP, отвечающая за преобразование объектов API Istio в информацию о конфигурации для распределенных прокси, а также направление входящего и исходящего трафика сервисной сетки.
2. Управляемый ЦС – это централизованный центр сертификации, отвечающий за предоставление SSL-сертификатов каждому из распределенных прокси, аутентификационную информацию и раздачу секретов.
3. Инструменты операций, ранее называвшиеся stackdriver, предоставляют управляемую точку приема для наблюдения и телеметрии, в частности, для мониторинга, отслеживания и регистрации данных, генерируемых каждым из прокси-серверов. Это позволяет операторам использовать панель наблюдения, позволяющую операторам визуально проверять свои сервисы и зависимости от сервисов, помогая внедрить лучшие практики SRE для мониторинга SLI и установления SLO.

</b></details>

<details>
<summary>Чем помогает Anthos Service Mesh?</summary><br><b>

Интеграция инструментов и технологий, составляющая сервисную сетку Anthos, обеспечивает значительные эксплуатационные преимущества для сред Anthos с минимальными дополнительными накладными расходами, например:

* Равномерная наблюдаемость — плоскость данных сообщает о взаимодействии между службами обратно в плоскость управления, создавая граф зависимости служб. Проверка трафика прокси-сервером вставляет заголовки для облегчения распределенной трассировки, сбора и составления отчетов журналов службы вместе с метриками уровня обслуживания (т. е. задержкой, ошибками, доступностью).
* Оперативная гибкость — предусмотрены детализированные элементы управления для управления потоками межсетевого (север-юг) и внутрисетевого (восток-запад) трафика.
* Безопасность на основе политик — политики могут применяться согласованно для различных протоколов и сред выполнения, поскольку служебные коммуникации защищены по умолчанию.

</b></details>

* Разделение трафика по разным версиям сервиса для канареечного или A/B-тестирования.
* Размыкание цепи для предотвращения каскадных сбоев.
* Внедрение ошибок для создания устойчивых и отказоустойчивых развертываний.
* Управление трафиком на основе HTTP-заголовка между отдельными службами или версиями.

</b></details>

<details>
<summary>Что такое Cloud Run для Anthos?</summary><br><b>

Это часть стека Anthos, который обеспечивает работу бессерверных контейнеров в Anthos, предлагая высокоуровневую платформу поверх кластеров K8s. Он создан с использованием Knative, оператора с открытым исходным кодом для K8s, который обеспечивает возможности бессерверного обслуживания приложений и обработки событий.

</b></details>

<details>
<summary>Как Cloud Run для Anthos упрощает работу?</summary><br><b>

Команды разработчиков платформ в организациях, которые хотят предложить разработчикам дополнительные инструменты для тестирования, развертывания и запуска приложений, могут использовать Knative для улучшения этих возможностей в Anthos как Cloud Run. Ниже приведены некоторые преимущества;

* Простая миграция с развертываний K8s. Без Cloud Run инженерам платформы приходится настраивать развертывание, обслуживание и объекты HorizontalPodAutoscalers (HPA) для балансировщика нагрузки и автоматического масштабирования. Если приложение уже обслуживает трафик, становится сложно изменить конфигурации или эффективно выполнить откат. Используя Cloud Run, все это управляется, поэтому манифест службы Knative описывает приложение, которое должно быть автоматически масштабировано и сбалансировано по нагрузке.
* Автомасштабирование — внезапный всплеск трафика может привести к сбою контейнеров приложений в K8s из-за перегрузки, поэтому для обслуживания большого объема трафика выполняется эффективное автоматическое автомасштабирование.
* Сеть — имеет встроенные возможности балансировки нагрузки и политики разделения трафика между несколькими версиями приложения.
* Релизы и развертывания — поддерживает понятие ревизий Knatibe API, которые описывают новые версии или различные конфигурации вашего приложения и канареечные развертывания путем разделения трафика.
* Мониторинг — наблюдение и запись таких показателей, как задержка, частота ошибок и количество запросов в секунду.

</b></details>

<details>
<summary>Перечислите и объясните три готовых примитива автомасштабирования высокого уровня, предлагаемых Cloud Run для Anthos, которые изначально не существуют в K8s.</summary><br><b>

* Быстрое автомасштабирование на основе запросов. Автомасштабирование по умолчанию отслеживает показатели запросов, что позволяет Cloud Run для Anthos плавно обрабатывать скачкообразный трафик.
* Контроль параллелизма — такие ограничения, как максимальное количество запросов в полете на контейнер, применяются, чтобы гарантировать, что контейнер не перегрузится и не выйдет из строя. Добавляются дополнительные контейнеры для обработки скачкообразного трафика и буферизации запросов.
* Масштабировать до нуля — если приложение какое-то время неактивно, Cloud Run масштабирует его до нуля, чтобы уменьшить занимаемое им пространство. В качестве альтернативы можно отключить масштабирование до нуля, чтобы предотвратить холодный запуск.

</b></details>
